# 前综述 02：模块化综述框架、NotebookLM 提示词与提问执行

## 概述

本 skill 用于文献检索、筛选、下载完成之后的模块化综述阶段。核心任务是：

1. 搭建问题导向的模块化综述框架
2. 为每个模块分配支撑文献
3. 生成可直接发送给 NotebookLM 的高质量提示词
4. 在用户完成 notebook 与资料准备、并明确授权后执行提问

这不是整篇综述自动写作 skill。

## 执行原则

- 模块框架必须围绕研究问题组织，而不是按作者列表堆砌
- 提示词必须服务于后续论文写作，而不是生成泛泛总结
- 若用户提供范例、手稿、已写段落或目标文风，提示词必须吸收这些风格约束
- 默认要求 NotebookLM 在正文中尽量给出 APA 风格文中引用
- 默认要求文末附参考文献列表
- 默认采用可见执行，不隐藏在后台
- 每轮回答后默认进入人工检查回路
- 如信息不足，最多只补问 1-2 个关键问题；能出初版时先出初版

## NotebookLM 适配原则

NotebookLM 更适合：

- 基于上传资料做 grounded 阅读
- 比较不同文献
- 围绕中心问题做整合
- 生成适合论文模块继续修改的文本

默认提示词应强调：

- 问题明确
- 模块边界清楚
- 不要逐篇摘要堆砌
- 围绕共识、差异、争议和空白组织材料
- 尽量生成适合论文使用的学术表达

## 引用要求

提示词默认要求 NotebookLM 同时提供：

1. 正文中的 APA 风格文中引用
2. 文末参考文献列表

如果用户要求利用已上传论文中的引文网络，可以追加“滚雪球式引用挖掘”，但必须区分：

- 当前上传文献直接支持
- 文中转引支持

不要把转引误写成已上传原文直接证明的内容。

## 工作流程

### Step 1：理解综述任务

先确认：

- 研究主题
- 研究对象
- 研究问题
- 研究意图
- 已确认文献清单或汇总表
- 用户的风格要求
- 用户是否希望启用滚雪球式引用挖掘

### Step 2：生成模块化综述框架

默认可包含：

- 研究背景 / 问题提出
- 概念界定
- 理论路径
- 经验研究进展
- 核心争议
- 中国 / 本土研究
- 研究空白

每个模块至少说明：

- 模块名称
- 模块作用
- 核心问题
- 对应文献
- 核心支撑文献
- 补充支撑文献

### Step 3：为每个模块分配文献

输出时应明确：

- 为什么某篇文献属于该模块
- 哪些文献是核心支撑
- 哪些文献是补充支撑
- 哪些文献可以跨模块复用

### Step 4：为每个模块生成 NotebookLM 提示词

每个模块至少生成三类 prompt：

1. 抽取型
2. 比较型
3. 写作型

默认 prompt 结构：

1. 任务目标
2. 回答范围
3. 输出结构
4. APA 文中引用要求
5. 文末参考文献列表要求
6. 额外限制

### Step 5：将提示词写成 NotebookLM 友好形式

除非用户另有要求，否则提示词应尽量包含：

- 不要泛泛概述
- 不要逐篇摘要堆砌
- 围绕中心问题整合文献
- 突出一致结论、差异、争议和空白
- 用尽量适合论文的学术表达
- 如提供了风格范例，尽量模仿其论述节奏、句式密度、概念组织方式和学术语气
- 如需要滚雪球式引用，允许基于已上传论文中的引文网络补充讨论，但必须明确区分直接依据与转引依据

### Step 6：明确停止点

在生成完框架、文献分配和提示词后，必须停止。

等待用户：

- 手动创建 NotebookLM notebook
- 手动上传对应资料
- 明确告知可以开始提问

在此之前不得自动调用 NotebookLM。

### Step 6.5：NotebookLM 资料上传准备

若用户希望自动上传资料，先检查：

- 目标 notebook 已由用户手动创建
- 当前日常 Chrome 登录态可用
- Web-access 或等效浏览器自动化通道健康可用
- 待上传文献在 Zotero 中已有本地 PDF

如果条件不满足，不直接进入上传动作。

### Step 6.6：NotebookLM 自动上传的高效路径

当用户明确授权自动上传时，优先使用已验证的路径：

1. 从 Zotero collection 精确筛出要上传的文献
2. 只选择真正存在的本地 PDF
3. 复用用户日常 Chrome 登录态
4. 激活目标 NotebookLM 页面
5. 对页面中的上传入口执行真实鼠标点击
6. 使用浏览器级文件选择器事件一次性注入 PDF 路径

### Step 6.7：批量上传优先于逐篇上传

优先：

- 先收集本次所有本地 PDF 路径
- 一次性注入多篇 PDF
- 上传后统一回查来源区中的文件名和来源数量

只有在多文件路径注入失败时，才降级为逐篇上传。

### Step 6.8：回查与完成判定

上传完成后，不要只以“文件选择器已触发”为成功标准。必须同时满足：

- 页面来源区出现新增的 PDF 文件名
- 来源数量与预期相符或明显增加

满足以上条件后，才可视为“资料已上传，可进入提问阶段”。

### Step 7：执行 NotebookLM 提问

只有在用户明确告知以下信息后，才进入执行阶段：

- notebook 已创建
- 对应资料已上传
- 可以开始提问

执行时：

1. 确认本次要提问的模块
2. 确认要使用哪条 prompt
3. 确认目标 notebook
4. 默认以可见窗口模式执行
5. 记录本轮模块、prompt 版本与 notebook

### Step 8：输出提问执行结果

至少说明：

- 对哪个模块发起了提问
- 使用了哪条提示词
- 面向哪个 notebook 执行
- 是否成功拿到回答
- 是否建议继续追问
- 本轮是否使用了风格约束
- 本轮是否启用了滚雪球式引用挖掘

默认进入人工审阅回路，等待用户评价是否达标。

## 输出结构

### 1. 模块化综述框架

- 模块名称
- 模块目标
- 核心问题
- 对应文献
- 核心支撑文献
- 补充支撑文献

### 2. 模块-文献对应关系

- 每个模块对应哪些文献
- 为什么这些文献属于该模块
- 哪些文献可跨模块复用

### 3. NotebookLM 提示词

每个模块至少给出：

- 抽取型 prompt
- 比较型 prompt
- 写作型 prompt

### 4. 提问执行结果

在获得用户授权并执行后，输出：

- 执行模块
- 使用提示词
- 执行状态

### 5. 资料上传结果

若本轮包含自动上传，还应额外输出：

- 目标 notebook
- 上传方式：`single-file` 或 `batch-file`
- 上传来源：Zotero collection 名称或条目范围
- 实际上传的 PDF 数量
- 页面回查结果
- 是否已可进入提问阶段

## 执行边界

- 不直接替用户写整篇综述成稿
- 不把模块框架写成作者列表堆砌
- 不生成没有明确模块目标的宽泛 prompt
- 不忽略 APA 文中引用与参考文献要求
- 在用户未完成 notebook 与资料准备前，不自动执行提问
- 在用户未明确授权前，不自动进入提问执行阶段
